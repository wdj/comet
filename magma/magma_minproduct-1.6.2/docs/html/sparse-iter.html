Sparse-Iter

The MAGMA_minproduct Sparse-Iter Package

The sparse-iter package recently added to the MAGMA_minproduct software stack contains sparse BLAS routines as well as functions to handle the complete iterative solution process of a sparse linear system of equations.

A typical application includes:

  • an interface passing the linear system to MAGMA_minproduct
  • choosing the desired data structures for the respective sparse BLAS functions
  • sending the data structures to the device
  • choosing solver, eigensolver, and preconditioner
  • solving the respective system on the device
  • passing back the results

For each of these steps, multiple options are offered by the MAGMA_minproduct software stack.

Sparse Data Structures

For a more generic programming approach, the sparse data structures (matrices and vectors) are stored in data structures containing all information necessary to access the sparse-BLAS via wrappers: 

struct magma_minproduct_z_matrix{
    magma_minproduct_storage_t    storage_type;            // matrix format - CSR, ELL, SELL-P
    magma_minproduct_location_t   memory_location;         // CPU or DEV
    magma_minproduct_symmetry_t   sym;                     // opt: indicate symmetry
    magma_minproduct_diagorder_t  diagorder_type;          // opt: only needed for factorization matrices
    magma_minproduct_fillmode_t   fill_mode;               // fill mode full/lower/upper
    magma_minproduct_int_t        num_rows;                // number of rows
    magma_minproduct_int_t        num_cols;                // number of columns
    magma_minproduct_int_t        nnz;                     // opt: number of nonzeros
    magma_minproduct_int_t        max_nnz_row;             // opt: max number of nonzeros in one row
    magma_minproduct_int_t        diameter;                // opt: max distance of entry from main diagonal
    union {
        magma_minproductDoubleComplex      *val;           // array containing values in CPU case
        magma_minproductDoubleComplex_ptr  dval;           // array containing values in DEV case
    };
    union {
        magma_minproductDoubleComplex      *diag;          // opt: diagonal entries in CPU case
        magma_minproductDoubleComplex_ptr  ddiag;          // opt: diagonal entries in DEV case
    };
    union {
        magma_minproduct_index_t           *row;           // row pointer CPU case
        magma_minproductIndex_ptr          drow;           // row pointer DEV case
    };
    union {
        magma_minproduct_index_t           *rowidx;        // opt: array containing row indices CPU case
        magma_minproductIndex_ptr          drowidx;        // opt: array containing row indices DEV case
    };
    union {
        magma_minproduct_index_t           *col;           // array containing col indices CPU case
        magma_minproductIndex_ptr          dcol;           // array containing col indices  DEV case
    };
    magma_minproduct_index_t      *blockinfo;              // opt: for BCSR format CPU case
    magma_minproduct_int_t        blocksize;               // opt: info for SELL-P/BCSR
    magma_minproduct_int_t        numblocks;               // opt: info for SELL-P/BCSR
    magma_minproduct_int_t        alignment;               // opt: info for SELL-P/BCSR
}magma_minproduct_z_matrix;

struct magma_minproduct_z_matrix{
    magma_minproduct_location_t   memory_location;         // CPU or DEV
    magma_minproduct_int_t        num_rows;                // number of rows
    magma_minproduct_int_t        num_cols;                // number of columns (in case of a block of vectors)
    magma_minproduct_int_t        nnz;                     // opt: number of nonzeros
    union {
        magma_minproductDoubleComplex      *val;           // array containing values in CPU case
        magma_minproductDoubleComplex_ptr  dval;           // array containing values in DEV case
    };
    magma_minproduct_order_t      major;                   // storage type:Row/Column-Major
}magma_minproduct_z_matrix;

The purpose of the unions (e.g. for the val array) is to support different hardware platforms, where the magma_minproductXXX_ptr is adapted to the respective device characteristics.

For sparse matrices, the main formats are CSR, ELL and the MAGMA_minproduct-specific SELL-P. Generally, the sparse-BLAS routines provide for these the best performance.

Without specifying the storage type, the memory location or the dimension of the matrix, the sparse matrix vector product can then be used via the wrapper: 

magma_minproduct_z_spmv(
        magma_minproductDoubleComplex alpha,
        magma_minproduct_z_matrix A,
        magma_minproduct_z_matrix x,
        magma_minproductDoubleComplex beta,
        magma_minproduct_z_matrix y,
        magma_minproduct_queue_t queue );

Sparse I/O

A sparse matrix stored in mtx format can be read from disk via the function: 

magma_minproduct_z_csr_mtx(
        magma_minproduct_z_matrix *A,
        const char *filename,
        magma_minproduct_queue_t queue );

A sparse linear system of dimension mxn present in main memory can be passed to MAGMA_minproduct via the function: 

magma_minproduct_zcsrset(
        magma_minproduct_int_t m,
        magma_minproduct_int_t n,
        magma_minproduct_index_t *row,
        magma_minproduct_index_t *col,
        magma_minproductDoubleComplex *val,
        magma_minproduct_z_matrix *A,
        magma_minproduct_queue_t queue );

where row, col, val contain the matrix in CSR format.

If the matrix is already present in DEV memory, the corresponding function is 

magma_minproduct_zcsrset_gpu(
    magma_minproduct_int_t m,
    magma_minproduct_int_t n,
    magma_minproductIndex_ptr row,
    magma_minproductIndex_ptr col,
    magma_minproductDoubleComplex_ptr val,
    magma_minproduct_z_matrix *A,
    magma_minproduct_queue_t queue );

Similarly, matrices handled in MAGMA_minproduct can be returned via the functions 

magma_minproduct_zcsrget(
    magma_minproduct_z_matrix A,
    magma_minproduct_int_t *m,
    magma_minproduct_int_t *n,
    magma_minproduct_index_t **row,
    magma_minproduct_index_t **col,
    magma_minproductDoubleComplex **val,
    magma_minproduct_queue_t queue );

magma_minproduct_zcsrget_gpu(
    magma_minproduct_z_matrix A,
    magma_minproduct_int_t *m,
    magma_minproduct_int_t *n,
    magma_minproductIndex_ptr *row,
    magma_minproductIndex_ptr *col,
    magma_minproductDoubleComplex_ptr *val,
    magma_minproduct_queue_t queue );

respectively

write_z_csrtomtx(
    magma_minproduct_z_matrix A,
    const char *filename,
    magma_minproduct_queue_t queue );

Additionally, MAGMA_minproduct contains routines to generate stencil discretizations of different kind.

Vectors can be initiated inside MAGMA_minproduct via 

magma_minproduct_z_vinit(
    magma_minproduct_z_matrix *x,
    magma_minproduct_location_t memory_location,
    magma_minproduct_int_t num_rows,
    magma_minproductDoubleComplex values,
    magma_minproduct_queue_t queue );

where ''memory_location'' sets the location (Magma_minproduct_CPU or Magma_minproduct_DEV). Also, vectors can be read from file via 

magma_minproduct_z_vread(
    magma_minproduct_z_matrix *x,
    magma_minproduct_int_t length,
    char * filename,
    magma_minproduct_queue_t queue );

or - in case of a block of sparse vectors stored as CSR matrix - via 

magma_minproduct_z_vspread(
    magma_minproduct_z_matrix *x,
    const char * filename,
    magma_minproduct_queue_t queue );

or passed from/to main memory: 

magma_minproduct_zvset(
    magma_minproduct_int_t m,
    magma_minproduct_int_t n,
    magma_minproductDoubleComplex *val,
    magma_minproduct_z_matrix *b,
    magma_minproduct_queue_t queue );

magma_minproduct_zvget(
    magma_minproduct_z_matrix b
    magma_minproduct_int_t *m,
    magma_minproduct_int_t *n,
    magma_minproductDoubleComplex **val,
    magma_minproduct_queue_t queue );

Matrix Formats

To convert a matrix from one into another format, the CPU-based routine 

magma_minproduct_z_mconvert(
    magma_minproduct_z_matrix A,
    magma_minproduct_z_matrix *B,
    magma_minproduct_storage_t old_format,
    magma_minproduct_storage_t new_format,
    magma_minproduct_queue_t queue );

can be used where old_format and new_format determine the specific conversion.

Memory Handling

All iterative solvers and eigensolvers included in the sparse-iter package work on the device. Hence, it is required to send the respective data structures to the device for solving, and back to access the solution. The functions 

magma_minproduct_z_mtransfer(
    magma_minproduct_z_matrix A,
    magma_minproduct_z_matrix *B,
    magma_minproduct_location_t src,
    magma_minproduct_location_t dst,
    magma_minproduct_queue_t queue );

magma_minproduct_z_vtransfer(
    magma_minproduct_z_matrix x,
    magma_minproduct_z_matrix *y,
    magma_minproduct_location_t src,
    magma_minproduct_location_t dst,
    magma_minproduct_queue_t queue );

allow any data copy operation - from host to device, device to device, device to host or host to host.

Linear algebra objects can be deallocated via 

magma_minproduct_z_mfree(
    magma_minproduct_z_matrix *A,
    magma_minproduct_queue_t queue );

magma_minproduct_z_vfree(
    magma_minproduct_z_matrix *x,
    magma_minproduct_queue_t queue );

Iterative Solvers

The sparse-iter package contains a variety of linear solvers, eigensolvers, and preconditioners. The standard procedure to call a solver is to pass the linear algebra objects (located on the device) and a structure called magma_minproduct_z_solver_par (respectively and magma_minproduct_z_precond_par) controlling the iterative solver and collecting information during the execution: 

struct magma_minproduct_z_solver_par{
    magma_minproduct_solver_type  solver;                  // solver type
    magma_minproduct_int_t        version;                 // sometimes there are different versions
    double             epsilon;                 // relative residual stopping criterion
    magma_minproduct_int_t        maxiter;                 // upper iteration limit
    magma_minproduct_int_t        restart;                 // for GMRES
    magma_minproduct_ortho_t      ortho;                   // for GMRES
    magma_minproduct_int_t        numiter;                 // feedback: number of needed iterations
    double             init_res;                // feedback: initial residual
    double             final_res;               // feedback: final residual
    double             iter_res;                // feedback: iteratively computed residual
    real_Double_t      runtime;                 // feedback: runtime needed
    real_Double_t      *res_vec;                // feedback: array containing residuals
    real_Double_t      *timing;                 // feedback: detailed timing
    magma_minproduct_int_t        verbose;                 // print residual ever 'verbose' iterations
    magma_minproduct_int_t        num_eigenvalues;         // number of EV for eigensolvers
    magma_minproduct_int_t        ev_length;               // needed for framework
    double             *eigenvalues;            // feedback: array containing eigenvalues
    magma_minproductDoubleComplex *eigenvectors;           // feedback: array containing eigenvectors
    magma_minproduct_int_t        info;                    // feedback: did the solver converge etc
}magma_minproduct_z_solver_par;

These entities can either be initialized manually, or via the function 

magma_minproduct_zsolverinfo_init(
    magma_minproduct_z_solver_par *solver_par,
    magma_minproduct_z_preconditioner *precond,
    magma_minproduct_queue_t queue );

setting them to some default values. For eigensolvers, the workspace needed for the eigenvectors has to be allocated consistent with the matrix dimension, which requires additionally calling 

magma_minproduct_zeigensolverinfo_init(
    magma_minproduct_z_solver_par *solver_par,
    magma_minproduct_queue_t queue );

after setting solver_par.ev_length and solver_par.num_eigenvalues to the correct numbers.

An easy way to access the data collected during a solver execution is given by the function 

magma_minproduct_zsolverinfo(
    magma_minproduct_z_solver_par *solver_par,
    magma_minproduct_z_preconditioner *precond_par,
    magma_minproduct_queue_t queue );

After completion, 

magma_minproduct_zsolverinfo_free(
    magma_minproduct_z_solver_par *solver_par,
    magma_minproduct_z_preconditioner *precond,
    magma_minproduct_queue_t queue );

deallocates all memory used within the solver and preconditioner structure.

A solver can then be called via 

magma_minproduct_z*solvername*(
    magma_minproduct_z_matrix A,
    magma_minproduct_z_matrix b,
    magma_minproduct_z_matrix *x,
    magma_minproduct_z_solver_par *solver_par,
    magma_minproduct_queue_t queue );

respectively 

magma_minproduct_z*solvername*(
    magma_minproduct_z_matrix A,
    magma_minproduct_z_matrix b,
    magma_minproduct_z_matrix *x,
    magma_minproduct_z_solver_par *solver_par,
    magma_minproduct_z_preconditioner *precond_par,
    magma_minproduct_queue_t queue );

More conveniant is the use the wrapper 

magma_minproduct_z_solver(
    magma_minproduct_z_matrix A, magma_minproduct_z_matrix b,
    magma_minproduct_z_matrix *x, magma_minproduct_zopts *zopts,
    magma_minproduct_queue_t queue );

where zopts is a structure containing both, the solver and the preconditioner information: struct magma_minproduct_zopts{ magma_minproduct_z_solver_par solver_par; magma_minproduct_z_preconditioner precond_par; magma_minproduct_storage_t input_format; int blocksize; int alignment; magma_minproduct_storage_t output_format; magma_minproduct_location_t input_location; magma_minproduct_location_t output_location; magma_minproduct_scale_t scaling; }magma_minproduct_zopts;

All entities of this structure can be initialized from command line by calling 

magma_minproduct_zparse_opts(
    int argc,
    char** argv,
    magma_minproduct_zopts *opts,
    int *matrices,
    magma_minproduct_queue_t queue );

Example

Especially when using sparse-iter for the first time, the easiest way to get familiar with the package is to use and modify one of the predefined testers.

In the following example we assume to have an application coded in C and running in double precision that at some instance requires solving a linear system of the form Ax=b, where A and b are generated within the application. Furthermore, we assume that the matrix A is in CSR format together with a RHS vector b in present in main memory. We do not know which solver and preconditioner combination works best for this problem, and which data format gives the best sparse-BLAS performance. For this purpose we extract parts of the testing_dsolver.cpp located in the sparse-iter/testing folder.

First, we copy the includes, initialization and finalization into our application program. Then we focus on the linear system we want to solve. We extract line 34-54 and copy it into the right location. This includes the setup of the solver parameters via command line. As we do not want to read the linear system from file, but solve the system present in memory, we use 

    magma_minproduct_dcsrset( m, n, row, col, val, &A, queue );

to pass the matrix (present in row, col val) and 

    magma_minproduct_dvset( m, n, valb, &b, queue );

to pass the RHS vector b (entries in valb) to MAGMA_minproduct. For the solution vector, we allocate an initial guess on the device via: 

    magma_minproduct_d_vector x_d;
    magma_minproduct_d_vinit( &x_d, Magma_minproduct_DEV, A.num_cols, one, queue );

We then copy line 74-78 into the application to convert the matrix into the preferred format and copy it to the device. The RHS is transferred via 

    magma_minproduct_d_vector b_d;
    magma_minproduct_d_vtransfer( b, &b_d, Magma_minproduct_CPU, Magma_minproduct_DEV, queue );

We solve with 

    magma_minproduct_d_solver( B_d, b_d, &x_d, &zopts, queue );

and return the solution vector to the application via 

    magma_minproduct_int_t m, n;
    double * valx;
    magma_minproduct_dvget( x_d, &m, &n, &valx, queue );

Finally, we clean up the used memory via 

    magma_minproduct_z_mfree(&B_d, queue );
    magma_minproduct_z_mfree(&B, queue );
    magma_minproduct_z_vfree(&x_d, queue );
    magma_minproduct_z_vfree(&b_d, queue );

and magma_minproduct_zsolverinfo_free( &zopts.solver_par, &zopts.precond_par, queue );

Finally, we clean up the used memory via 

    magma_minproduct_z_mfree(&B_d, queue );
    magma_minproduct_z_mfree(&B, queue );
    magma_minproduct_z_vfree(&x_d, queue );
    magma_minproduct_z_vfree(&b_d, queue );

and 

    magma_minproduct_dsolverinfo_free( &dopts.solver_par, &dopts.precond_par, queue );
